# SDK Usage Guide

This page is intentionally practical. It is not the conceptual overview of FALCON Connect and it is not the full protocol reference. It is the page you reach for when you want to wire the SDK into a real application and you want examples that match the current exported API.

## Effect-first API

The SDK is **Effect-based**: protocol types use Effect `Schema`, HTTP clients return `Effect` values, and crypto helpers are `Effect` programs. Import from `@falcon/sdk/effect` or the package root (`import "@falcon/sdk"` re-exports the same symbols).

See `packages/sdk/CHANGELOG.md` for breaking changes when upgrading from older releases.

### Migration quick reference (from legacy Promise clients)

| Old pattern | Current pattern |
|-------------|-------------------|
| `createFalconConnectTargetClient` | `makeFalconConnectTargetService` or `FalconConnectTargetService` + `FalconConnectTargetConfig` |
| `createFalconConnectSourceClient` | `makeFalconConnectSourceService` or `FalconConnectSourceService` + `FalconConnectSourceConfig` |
| Zod `*Schema` from `protocol` | Effect `Schema` exports from `@falcon/sdk/effect` |
| `createFalconAppAuthHeaders`, … | `createFalconAppAuthHeaders`, `verifyFalconAppRequestEffect`, … in `effect/crypto` |
| Await client methods directly | `Effect.runPromise(client.method(...))` (or another Effect runner) |

### Minimal Effect wiring (illustrative)

```ts
import { Effect, Layer } from "effect";
import {
  type AppId,
  type IntentToken,
  FalconConnectTargetConfig,
  FalconConnectTargetService,
} from "@falcon/sdk/effect";

const configLayer = Layer.succeed(FalconConnectTargetConfig, {
  baseUrl: new URL("https://connect.falcon.example"),
  appId: "orders" as AppId,
  keyId: "orders-key-1",
  privateJwk: JSON.parse(process.env.FALCON_PRIVATE_JWK!),
});

const program = Effect.gen(function* () {
  const target = yield* FalconConnectTargetService;
  return yield* target.resolveInstallIntent("intent-token-here" as IntentToken);
});

// run with your preferred Effect runner and provide configLayer
```

The sections below walk through the same flows using **`Effect.runPromise`** on the service factories. Compare these snippets to `apps/demo-01` and `apps/demo-02`, which implement the source-side and target-side flows end to end.

The SDK is split into three parts:

- a source-app client for initiating connections, minting runtime tokens, and updating connection status when your product needs to pause, resume, or revoke from the source side
- a target-app client for resolving install intents, submitting consent decisions, looking up an existing incoming connection by identity keys, verifying runtime tokens, and updating connection status from the target side
- lower-level crypto and UI helpers for custom integrations

All examples below are written as plain TypeScript so they are easy to adapt to whatever HTTP framework you are using.

## Installation

Install the SDK in the partner application:

```bash
bun add @falcon/sdk
```

The SDK expects that your application already has:

- a FALCON base URL
- a trusted app id
- an active key id
- a private JWK string or object for signing requests to FALCON

In most applications you will load those values from environment variables or your application config layer.

If you need a fresh Ed25519 JWK pair locally, run:

```bash
bun run gen:jwk -- --mode trusted-app --key-id orders-key-1
```

That helper prints a private JWK for `FALCON_PRIVATE_JWK` and the public JWK you should register in Falcon.

## Basic Configuration

The smallest useful setup is to build one **service** on the source side or the target side with `makeFalconConnectSourceService` / `makeFalconConnectTargetService` (or the `Layer`-based `FalconConnect*Service` classes) and reuse it rather than re-creating it for every request.

```ts
import { Effect } from "effect";
import {
  AppId,
  makeFalconConnectSourceService,
  makeFalconConnectTargetService,
} from "@falcon/sdk/effect";

const falconBaseUrl = new URL("https://connect.falcon.example");
const appId = AppId.make("orders");
const keyId = "orders-key-1";
const privateJwk = JSON.parse(process.env.FALCON_PRIVATE_JWK!);

export const falconSource = makeFalconConnectSourceService({
  baseUrl: falconBaseUrl,
  appId,
  keyId,
  privateJwk,
});

export const falconTarget = makeFalconConnectTargetService({
  baseUrl: falconBaseUrl,
  appId,
  keyId,
  privateJwk,
});

// Example: run an effect to a Promise at the edge of your app
export async function runSource<A>(effect: Effect.Effect<A>) {
  return Effect.runPromise(effect);
}
```

You would normally create **one** of these service instances per app role. A source-only app usually needs only the source service. A target-only app usually needs only the target service. An app that can act in both roles may create both.

Each method on the service returns an `Effect`. Use `Effect.runPromise` (or `Effect.runFork`, a custom runtime, etc.) where you need async/await or framework handlers.

## Source App: Start an Installation Flow

The source app starts the relationship by creating an install intent.

```ts
import { Effect } from "effect";
import { falconSource } from "./falcon";

export async function startConnectionFlow() {
  const result = await Effect.runPromise(
    falconSource.createInstallIntent({
      targetAppId: "invoices",
      falconSubjectId: "usr_123",
      organizationId: "org_456",
      requestedScopes: ["invoices:read", "invoices:search"],
      sourceReturnUrl: new URL("https://orders.example.com/settings/falcon/callback"),
      expiresInSeconds: 900,
    }),
  );

  return result.connectUrl;
}
```

In a browser-driven flow, the usual next step is to redirect the user to `result.connectUrl`.

For example, in an Express-style handler:

```ts
export async function connectInvoicesHandler(req: Request, res: Response) {
  const connectUrl = await startConnectionFlow();
  res.redirect(connectUrl);
}
```

The returned object also includes `intentId`, `intentToken`, and `expiresAt`, which are useful for logging and debugging. The source app normally does not need to persist the install token itself once the redirect is issued.

## Source App: Handle the Callback

After the target app collects consent, FALCON returns a callback URL to the target app, which then redirects the user back to your source app. The source SDK includes a helper to parse that callback state.

```ts
import { falconSource } from "./falcon";

export function handleFalconCallback(url: string) {
  const callback = falconSource.parseInstallCallback(url);

  if (callback.status === "approved" && callback.connectionId) {
    return {
      ok: true,
      connectionId: callback.connectionId,
      intentId: callback.intentId,
    };
  }

  return {
    ok: false,
    status: callback.status,
    reason: callback.reason,
    intentId: callback.intentId,
  };
}
```

A typical source app persists the `connectionId` in its own integration table as soon as the callback reports approval.

## Source App: Mint a Runtime Connection Token

Once the connection exists, the source app uses the stored `connectionId` to mint a short-lived FALCON token whenever it wants to call the target app.

```ts
import { Effect } from "effect";
import { falconSource } from "./falcon";

export async function getConnectionToken(connectionId: string) {
  const result = await Effect.runPromise(
    falconSource.issueConnectionAccessToken({
      connectionId,
      expiresInSeconds: 300,
    }),
  );

  return result.token;
}
```

If you also want the decoded claim set FALCON used, the SDK already returns that in `result.claims`.

```ts
import { Effect } from "effect";

const result = await Effect.runPromise(
  falconSource.issueConnectionAccessToken({
    connectionId: "conn_123",
  }),
);

console.log(result.claims.targetAppId);
console.log(result.claims.scopes);
console.log(result.expiresAt);
```

## Source App: Call the Target App

FALCON Connect does not define your business API shape, so the exact outbound request is up to you. A common pattern is to place the FALCON token in an authorization-style header.

```ts
export async function fetchInvoices(connectionId: string) {
  const token = await getConnectionToken(connectionId);

  const response = await fetch("https://invoices.example.com/api/invoices/search", {
    method: "POST",
    headers: {
      "content-type": "application/json",
      authorization: `Bearer ${token}`,
    },
    body: JSON.stringify({
      orderId: "ord_123",
    }),
  });

  if (!response.ok) {
    throw new Error(`Invoice request failed with ${response.status}`);
  }

  return response.json();
}
```

## Source and Target: Update Connection Status

Either party on a directional connection may change its status through the SDK. The authenticated app must be the connection’s **source** or **target**; FALCON rejects callers that are not parties to the connection.

Use `active` to resume a paused connection, `paused` to suspend it, or `revoked` to end it. An optional `reason` is stored for audit (for `revoked`, the server may substitute a default if omitted). For how paused and revoked differ at runtime, see [Security and Operations](../security-and-operations).

```ts
import { Effect } from "effect";
import { falconSource } from "./falcon";

export async function pauseFromSource(connectionId: string) {
  return Effect.runPromise(
    falconSource.updateConnectionStatus({
      connectionId,
      status: "paused",
      reason: "Paused by user in source app settings",
    }),
  );
}
```

The target service exposes the same method:

```ts
import { Effect } from "effect";
import { falconTarget } from "./falcon";

export async function revokeFromTarget(connectionId: string) {
  return Effect.runPromise(
    falconTarget.updateConnectionStatus({
      connectionId,
      status: "revoked",
      reason: "Integration removed in target admin",
    }),
  );
}
```

## Target App: Resolve the Install Intent

When the user lands on the target app's connect route, the first FALCON-specific step is to extract the install token from the query string and resolve it.

```ts
import { IntentToken } from "@falcon/sdk/effect";
import { Effect } from "effect";
import { falconTarget } from "./falcon";

export async function resolveIncomingIntent(requestUrl: string) {
  const url = new URL(requestUrl);
  const intentToken = url.searchParams.get("falcon_connect_intent");

  if (!intentToken) {
    throw new Error("Missing falcon_connect_intent query parameter");
  }

  const intent = await Effect.runPromise(
    falconTarget.resolveInstallIntent(IntentToken.make(intentToken)),
  );

  return { intentToken, intent };
}
```

Keep both values. The resolved intent object does **not** replace the original token. The current `approveInstallIntent` helper expects both the resolved intent and the original `intentToken`.

## Target App: Build the Consent UI State

The target app owns the consent screen, but the SDK can prepare the scope-selection model so your UI logic does not have to rebuild the required/system rules by hand.

```ts
import { buildConsentSelection } from "@falcon/sdk/effect";

export function buildConsentViewModel(
  intent: Awaited<ReturnType<typeof resolveIncomingIntent>>["intent"],
) {
  return buildConsentSelection(intent).map((scope) => ({
    id: scope.name,
    title: scope.displayName,
    description: scope.description,
    checked: scope.selected,
    disabled: scope.locked,
  }));
}
```

The important behavior is that `locked` already reflects FALCON's rules for required and system scopes.

## Target App: Approve the Install

If the user approves the request, you can either use the higher-level `approveInstallIntent` helper or the lower-level `submitInstallIntentDecision` helper.

The higher-level helper is the easiest path when you only need to submit an approval:

```ts
import { IntentToken } from "@falcon/sdk/effect";
import { Effect } from "effect";
import { falconTarget } from "./falcon";

export async function approveConnection(input: {
  intentToken: string;
  intent: Awaited<ReturnType<typeof resolveIncomingIntent>>["intent"];
  selectedScopeNames: string[];
}) {
  const result = await Effect.runPromise(
    falconTarget.approveInstallIntent({
      intentToken: IntentToken.make(input.intentToken),
      intent: input.intent,
      selectedScopeNames: input.selectedScopeNames,
    }),
  );

  return result.redirectUrl;
}
```

The `selectedScopeNames` array should contain only the optional scopes the user kept enabled. Required and system scopes are automatically preserved by the helper.

## Target App: Deny the Install

If the user denies the request, use `submitInstallIntentDecision` directly:

```ts
import { Effect } from "effect";
import { falconTarget } from "./falcon";

export async function denyConnection(intentToken: string, deniedReason?: string) {
  const result = await Effect.runPromise(
    falconTarget.submitInstallIntentDecision({
      intentToken,
      approved: false,
      grantedScopes: [],
      deniedReason,
    }),
  );

  return result.redirectUrl;
}
```

The returned `redirectUrl` is what you should use to send the user back to the source app once the decision is complete.

## Target App: Find an Incoming Connection

When your UI already knows which **source app**, **FALCON subject**, and **organization** the user is acting in—but you do not have a `connectionId` handy—you can ask FALCON for the best-matching connection **to your app as the target**. Typical uses include settings or dashboards that list “connections from App X for this workspace.”

The request is signed as the target app. FALCON returns `null` when no row matches. When several candidates exist, the server prefers an **active** connection, then a **paused** one, then the most recently updated among the latest matches.

```ts
import { Effect } from "effect";
import { falconTarget } from "./falcon";

export async function loadIncomingFromOrders(params: {
  sourceAppId: string;
  falconSubjectId: string;
  organizationId: string;
}) {
  const connection = await Effect.runPromise(
    falconTarget.findIncomingConnection({
      sourceAppId: params.sourceAppId,
      falconSubjectId: params.falconSubjectId,
      organizationId: params.organizationId,
    }),
  );

  if (!connection) {
    return { state: "none" as const };
  }

  return { state: "found" as const, connection };
}
```

## Target App: Verify a Runtime Token

At runtime, the target app usually extracts the FALCON JWT from the incoming request and verifies it through the target client.

```ts
import { Effect } from "effect";
import { falconTarget } from "./falcon";

export async function verifyIncomingFalconToken(token: string) {
  const verification = await Effect.runPromise(
    falconTarget.verifyConnectionToken({
      token,
      allowIntrospectionFallback: true,
    }),
  );

  return verification;
}
```

The return shape depends on whether verification succeeded locally or had to fall back to FALCON introspection.

If verification succeeds locally, the result contains the decoded JWT claims:

```ts
const verification = await verifyIncomingFalconToken(token);

if (verification.mode === "local") {
  console.log(verification.result.connectionId);
  console.log(verification.result.scopes);
}
```

If verification falls back to introspection, the result is an introspection payload instead:

```ts
const verification = await verifyIncomingFalconToken(token);

if (verification.mode === "introspection") {
  if (!verification.result.active || !verification.result.connection) {
    throw new Error(`Connection inactive: ${verification.result.reason}`);
  }

  const grantedScopes = verification.result.connection.grantedScopes
    .filter((scope) => scope.selected)
    .map((scope) => scope.name);

  console.log(grantedScopes);
}
```

That difference is important enough that your middleware should branch on `verification.mode` rather than assuming a single result shape.

## Target App: Example Middleware Shape

Here is a framework-neutral example of how you might wrap the runtime verification step before serving a protected endpoint:

```ts
import { Effect } from "effect";
import { falconTarget } from "./falcon";

export async function requireFalconConnection(request: Request) {
  const authorization = request.headers.get("authorization");

  if (!authorization?.startsWith("Bearer ")) {
    throw new Error("Missing FALCON bearer token");
  }

  const token = authorization.slice("Bearer ".length);
  const verification = await Effect.runPromise(
    falconTarget.verifyConnectionToken({
      token,
      allowIntrospectionFallback: true,
    }),
  );

  if (verification.mode === "local") {
    return {
      connectionId: verification.result.connectionId,
      sourceAppId: verification.result.sourceAppId,
      targetAppId: verification.result.targetAppId,
      scopes: verification.result.scopes,
      falconSubjectId: verification.result.falconSubjectId,
      organizationId: verification.result.organizationId,
    };
  }

  if (!verification.result.active || !verification.result.connection) {
    throw new Error(`Inactive FALCON connection: ${verification.result.reason}`);
  }

  return {
    connectionId: verification.result.connection.id,
    sourceAppId: verification.result.connection.sourceAppId,
    targetAppId: verification.result.connection.targetAppId,
    scopes: verification.result.connection.grantedScopes
      .filter((scope) => scope.selected)
      .map((scope) => scope.name),
    falconSubjectId: verification.result.connection.falconSubjectId,
    organizationId: verification.result.connection.organizationId,
  };
}
```

## Lower-Level Helper: Manual FALCON Request Signing

Most applications should stay at the source and target client level. If you need lower-level control, the SDK also exports request-signing helpers.

For example, to sign a FALCON request manually:

```ts
import { createFalconAppAuthHeaders } from "@falcon/sdk/effect";
import { Effect } from "effect";

export async function buildSignedHeaders(body: unknown) {
  return Effect.runPromise(
    createFalconAppAuthHeaders({
      appId: "orders",
      keyId: "orders-key-1",
      privateJwk: JSON.parse(process.env.FALCON_PRIVATE_JWK!),
      method: "POST",
      url: "https://connect.falcon.example/v1/install-intents",
      body: JSON.stringify(body),
    }),
  );
}
```

This is useful if you want to integrate with a custom HTTP layer while still matching FALCON's expected signature format.

## Lower-Level Helper: Consent Selection Normalization

If you are rendering a custom UI and want to normalize the selected scope set yourself, the SDK also exports helpers for that:

```ts
import { buildConsentSelection, normalizeGrantedScopes } from "@falcon/sdk/effect";

const selection = buildConsentSelection(intent, userSelectedScopeNames);
const grantedScopes = normalizeGrantedScopes(intent, userSelectedScopeNames);
```

`buildConsentSelection` is helpful when you need a richer UI model. `normalizeGrantedScopes` is helpful when you just need the final list of scope names to submit back to FALCON.

## A Complete Minimal Pair

If you want the shortest end-to-end summary of how the SDK pieces fit together, it looks like this:

1. Source app creates an install intent and redirects the user.
2. Target app resolves the install token and renders consent.
3. Target app approves or denies and redirects back.
4. Source app stores the resulting `connectionId`.
5. Source app later mints a runtime token with that `connectionId`.
6. Target app verifies the token locally and introspects on fallback.

Optional lifecycle steps you may need in production:

7. Target app looks up a connection with `findIncomingConnection` when identity context is known but `connectionId` is not.
8. Source or target app updates status with `updateConnectionStatus` when the product should pause, resume, or revoke outside the operator dashboard.

The first six steps are the minimal install-and-runtime path. Steps 7 and 8 are the usual extensions when you manage connection state from partner apps. Everything else in this guide is detail around those steps.
