# Architecture

## What Falcon Connect Stores

Falcon Connect persists:

- trusted app manifests
- trusted app signing keys
- target-app scope declarations
- install intents
- directional connections
- per-connection scope grants
- audit events
- used app-request nonces for replay protection

## What Falcon Connect Does Not Store

Falcon Connect does not store or proxy partner business payloads. App A still calls App B directly at App B's own data API base URL.

## Directionality

Connections are one-way.

If App A is connected to App B, App A can call App B. App B cannot call App A unless a second connection is installed separately.

## Subject Model

V1 uses:

- `falconSubjectId`
- `organizationId`

Those values are supplied by trusted apps today and are intentionally shaped so Falcon Auth can become the canonical issuer later without breaking the SDK contract.

## Consent Rules

- target apps declare scopes
- Falcon system scopes are always included
- required scopes are locked on
- optional scopes are toggleable
- the final granted set is stored on the connection record

## Runtime Verification

Falcon signs short-lived connection access tokens with its signing JWK.

Target apps:

1. verify locally against Falcon JWKS
2. introspect with Falcon when local verification is insufficient

That gives the normal path low latency while keeping revocation authoritative.
